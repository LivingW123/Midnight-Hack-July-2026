Sealed how it's secured

The privacy boundary, protocol by protocol

What the chain can and cannot learn

Every claim below points at the line of code that enforces it — a privacy system you cannot audit is a privacy system you cannot trust.

01

Hiding & binding commitments

A bid never goes on-chain. What goes on-chain is persistentCommit([amount, bundleMask], nonce) — a commitment built from the Compact standard library, with a fresh 32-byte nonce drawn per bid.

  • Hiding — the commitment reveals nothing about the amount. Two bids of the same value produce completely different commitments, because the nonce differs.
  • Binding — a bidder cannot later "open" the commitment to a different number. The reveal circuit recomputes the commitment from the witness and compares.
  • The nonce is written to local disk before the transaction is sent, so a crash between commit and reveal can never strand an unopenable bid.
const commitment = bidCommitment(amount, bundle, bidNonce());
assert(bids.lookup(id) == commitment, "does not open the sealed bid");

02

Compiler-enforced declassification

This is the guarantee that does the most work, and it is the one people find hardest to believe: the Compact compiler refuses to build a contract that leaks a private value by accident. Any path from a witness to the ledger must be marked disclose(...) explicitly.

That is not a lint. It is a compile error, and we hit it building this project — the first version of the constructor assigned the format parameter straight to the ledger:

Exception: auction-house.compact line 20 char 10:
  potential witness-value disclosure must be declared but is not:
    witness value potentially disclosed:
      the value of parameter fmt of the constructor
    nature of the disclosure:
      ledger operation might disclose the witness value

Privacy here is therefore not a property of careful coding. It is a property the toolchain will not let you violate silently — you have to write the word disclose and mean it.

03

Proving a fact without revealing the value

The Dutch auction is where zero knowledge earns its keep. A bidder proves their sealed reservation price clears the public clock price — and the circuit contains no disclose() on that amount anywhere.

assert(bids.lookup(id) == bidCommitment(amount, bundle, bidNonce()));
// private >= public. Nothing about `amount` is declassified.
assert(amount >= currentPrice, "reservation price is below the current price");
winner = id;
highestBid = currentPrice;  // the CLOCK price is public; the bid is not

The chain learns exactly one bit that it needed: someone was willing to pay at least the current price. It does not learn how much more they would have paid — so the winner's true demand survives the sale, and the seller cannot price-discriminate against them next time.

04

Local proving — secrets never cross the network

Witness functions are called by Midnight.js during proof generation, on this machine. Their return values become private circuit inputs and are never serialised into a transaction.

  • The proof server runs at http://127.0.0.1:6300 — loopback, not a hosted service.
  • Secret keys, bid amounts, nonces and schedule secrets live in a local identity store, not in contract state.
  • Private state at rest goes through an encrypted store with a password provider; the default is a devnet placeholder and PRIVATE_STATE_PASSWORD overrides it for anything real.

src/house-api.ts — witnesses(), createHouseProviders

05

Domain-separated pseudonymous identity

An on-chain identity is persistentHash([pad(32, "sealed:pk:"), secretKey]). The domain tag matters: it means a hash from this application can never be confused with, or replayed into, a hash built for another purpose — even one using the same secret.

Ownership checks compare a freshly derived key against the stored owner, so "only the seller may close" is enforced by a proof of key possession rather than by a transaction sender field.

assert(publicKey(localSecretKey()) == owner, "only the seller can close bidding");
06

Mechanism integrity

Privacy is not enough on its own — a sealed auction also has to be provably fair.

  • Candle: the secret end-index is committed in the constructor, before a single bid exists. The seller physically cannot move the close to exclude a bid they dislike, because openSchedule verifies the opening against that commitment.
  • Combinatorial: lockAllocation enumerates all five partitions of the three lots inside the circuit and takes the maximum. The winning allocation is proven revenue-optimal, not asserted by the seller.
  • Batch: uniform clearing at the lowest winning bid removes the incentive to snipe — everyone pays the same price regardless of arrival time.
  • Replay & double-bid: map membership makes one sealed bid per identity per auction structural, not a convention.
  • Phase guards: every circuit asserts its phase, so a late bid or an early reveal cannot prove at all.
07

Fraud detection that cannot become a back door

Shill detection normally works by reading bid amounts and increments. Here that is impossible — the values are sealed from us too. So the detector scores only metadata the ledger already publishes: arrival order, first-seen timing, whether a sealed bid was ever opened, and cross-auction overlap.

This is a stronger position than it first appears. A fraud system that needs plaintext bids is a fraud system that becomes a surveillance tool the moment it is compromised. This one has nothing extra to leak — its input is public by construction, and its BidEvent type carries no amount field at all.

✓

Verified, not asserted

The privacy claim is tested adversarially rather than assumed. The e2e suites flatten everything the indexer exposes — raw serialised contract state plus the parsed ledger — into one string, and search it for each bid amount in three encodings (big-endian hex, little-endian hex, and decimal).

  • Amounts are chosen so their byte encodings cannot appear by coincidence.
  • In the first-price suite the eventual winner reveals first, so every later reveal is a losing reveal — the worst case for leakage.
  • The Dutch suite asserts the winner's reservation price is absent from chain state both before the claim and after winning.
npm run test:e2e     # sealed first-price: correctness + privacy
npm run test:house   # dutch, batch, candle, combinatorial
npm run test:intel   # detector calibration, incl. false-positive cases
